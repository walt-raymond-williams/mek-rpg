# MekHQ Contract Market Probe Plan

Status: issue `#69` historical write-side validation plan, updated by issue `#112` after local `contracts.accept` implementation.

Purpose: define the narrow conditions under which MEK-RPG could ask the MegaMek workspace to prototype a MekHQ-owned contract-market accept/decline command.

## Decision

Contract-market accept/decline was a good write-side bridge probe by stable contract offer id. Since this plan was written, the local MegaMek/MekHQ workspace implemented guarded contract acceptance as `POST /campaign/command/contracts/accept`. Keep this document as historical design context and as future guidance for contract decline or refusal cases.

Manual MekHQ UI accept/decline, save, and read-only re-import remains a valid fallback workflow. For acceptance, the mature target now exists locally: an explicit MekHQ-owned command endpoint with dry-run/preflight, stable guard fields, prompt policy, command result metadata, and live reread verification. MEK-RPG must not implement direct `.cpnx`, `.cpnx.gz`, or XML edits or a broad `apply-pending-action` command.

## Preconditions

For any future contract decline command, do not create an implementation issue until all of these are true:

- A MekHQ-owned read-only export or source-backed sample identifies contract-market offers with stable ids or stable selectors.
- The offer selector includes guard fields: campaign id, campaign date, offer id, contract name, employer, target system if available, deadline or start date, contract type, and market/source owner.
- MekHQ source work identifies the exact accept and decline methods, including where market removal and contract creation happen.
- AtB and StratCon prompt/dialog behavior is mapped, including start prompts, employer liaison or opponent generation, faction-standing messages, facility rental prompts, and any contract automation prompts.
- A noninteractive prompt policy exists: refuse by default unless every required prompt has an explicit safe default or the command is scoped to no-prompt offers only.
- Disposable MekHQ save validation is available before any real campaign use.
- Live reread or saved re-import fields can prove success or failure without trusting MEK-RPG Markdown alone.

## Pending Action Shape

A future contract-market pending item should remain a normal `pending-mekhq-actions.md` entry with type `contract`. For command planning, it should include these details in existing fields:

- `MekHQ target ids`: campaign id and contract offer id; scenario id remains `Unknown` until generated by MekHQ.
- `Current imported baseline`: campaign date, market/export timestamp, offer id, offer name, employer, deadline/start date, current active contract count, finance balance if exported, and contract-market offer count.
- `Proposed MekHQ action`: accept or decline one specific contract-market offer by id.
- `Manual application checklist`: points to MekHQ UI only when no source-backed command exists or the command is unavailable, blocked, refused, or unverifiable.
- `Confirmation needed from next import`: the fields listed below for accept or decline.
- `Blockers or discrepancy notes`: missing stable id, duplicate guard fields, prompt required, unsupported helper field, or re-import mismatch.

Do not use table row number, display name alone, employer alone, or date alone as the selector.

## Accept Confirmation

After an accepted contract offer is applied in MekHQ and the campaign is saved, MEK-RPG needs a read-only import/export that confirms:

- same MekHQ campaign id and expected baseline date or accepted date
- contract offer id no longer appears in the contract market, or is explicitly marked consumed
- accepted contract appears in active missions/contracts with stable id
- name, employer, target system, deadline/start date, and terms match guard fields closely enough to detect the selected offer
- advance/transport/payment finance transactions appear when MekHQ creates them
- scenario or StratCon state appears if MekHQ generates it during acceptance
- reports include relevant contract acceptance messages if exported
- no unresolved prompt or partial-accept state remains

If any confirmation field is absent from the current helper/export, leave the pending item `blocked` or `imported` pending review; do not mark it `resolved` only because MEK-RPG intended the action.

## Decline Confirmation

Decline may have fewer side effects than accept, but it still changes market state and can still be risky without a stable selector.

After a declined contract offer is applied in MekHQ and the campaign is saved, MEK-RPG needs a read-only import/export that confirms:

- same MekHQ campaign id and expected baseline date or declined date
- selected offer id no longer appears in the contract market, or is explicitly marked declined/rejected if MekHQ retains history
- no new active contract with matching guard fields was created
- contract-market offer count or market status changed as expected
- reports include relevant decline/rejection messages if exported
- no unresolved prompt or partial-decline state remains

If MekHQ decline behavior only removes a row without durable history, the command needs stronger guard fields and before/after offer counts to avoid confusing the selected offer with a market refresh.

## Refusal Policy

A future command must refuse rather than guess when:

- the offer id is missing, duplicated, or not stable across save/export cycles
- guard fields do not match the expected name, employer, deadline/start date, type, or campaign id
- the target offer is absent from the latest checkpoint
- campaign date or save identity differs from the pending baseline and the user has not explicitly re-queued the action
- AtB/StratCon or campaign-option prompts are required and no safe explicit policy exists
- accepting would trigger scenario generation, facility rental, liaison/opponent creation, faction-standing changes, or other side effects that the command cannot confirm
- decline behavior is not source-confirmed
- saved re-import cannot prove whether the action succeeded

Refusal should leave the pending item open with a specific blocker. It should not edit the MekHQ save or silently fall back to a different selector.

## Disposable Validation Sequence

Before any real campaign use, validate in disposable MekHQ data:

1. Export a checkpoint with at least two contract-market offers, preferably including duplicate-like names or employers.
2. Queue one `contract` pending item in MEK-RPG with stable offer id and guard fields.
3. Run the MekHQ-owned command in dry-run mode, if available, and confirm it reports the exact target and side effects.
4. Run accept in disposable data and save through MekHQ-owned code.
5. Re-import or export the saved checkpoint and reconcile the pending item using the accept confirmation fields.
6. Repeat decline in a fresh disposable baseline.
7. Repeat one refusal scenario for missing id, mismatched guard field, and prompt-required offer.
8. Confirm no raw MekHQ saves, raw XML, protected A Time of War source text, or secrets are committed.

## MegaMek Workspace Issue Candidate

Only create this as a MegaMek-side implementation issue after read-only export support or a source-backed fixture proves stable selectors.

Title:

```text
Prototype MekHQ contract-market accept/decline by stable offer id
```

Goal:

```text
Prototype a MekHQ-owned command or helper that accepts or declines one contract-market offer by stable id in disposable campaign data, refuses when prompt policy or guard fields are unsafe, saves through MekHQ-owned logic, and supports read-only re-import confirmation.
```

Acceptance criteria:

- Loads a disposable MekHQ campaign through supported MekHQ code.
- Selects one contract-market offer by stable id plus guard fields, not UI row index.
- Supports dry-run reporting of target, side effects, and prompt blockers.
- Accepts a no-prompt or explicitly policy-covered offer and saves through MekHQ-owned save logic.
- Declines an offer only after source-confirming decline semantics and market removal/history behavior.
- Refuses missing ids, duplicate/mismatched guard fields, unsupported prompts, changed baseline, and unverifiable results.
- Provides read-only export fields that prove accept/decline success after save.
- Does not edit raw `.cpnx`, `.cpnx.gz`, or XML payloads directly.

## Non-Goals

- No MEK-RPG-side raw save/XML write command.
- No direct save/XML editing.
- No broad `apply-pending-action` bridge.
- No unrelated purchase/sale, hiring, repair/logistics, tactical-result, finance, or day-advance automation in this specific contract probe.
- No real campaign use before disposable validation and saved re-import confirmation.
